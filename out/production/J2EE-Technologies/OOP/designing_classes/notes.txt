    Designing Classes in Java

    Most devs fails here because they start with:
    "What fields do I need?"

    Strong engineers start with:
    "What responsibility should this class own -- and what should it refuse to do?"

    🧠 1. The core mindset shift
        Weak thinking (implementation-first)
        "I need a class with name, balance, and setters"
        Strong thinking (responsibility-first)
        "This class should ensure money is always valid and only changes through safe operations"

    🔥 2. The Rule of Class Design
           A class should have one reason to change

           Not: one class = many features BUT one class = one responsibility boundary

    🧠 8. Class design checklist (use this in interviews)

    Before writing a class, ask:

    1. What is this class RESPONSIBLE for?
        NOT what fields it has
    2. What rules must NEVER be broken?
        invariants
    3. Who is allowed to change state?
        only the class itself?
    4. What should be hidden?
        internal fields
        unsafe setters
    5. What behavior belongs here vs elsewhere?
        payment logic? maybe service
        state logic? belongs in domain object

    When designing classes, I focus on responsibility rather than fields. A class should own its invariants and
    enforce valid state transitions internally. Instead of exposing setters and raw data, I design behavior-driven
    classes that protect their own consistency and only expose safe operations.
