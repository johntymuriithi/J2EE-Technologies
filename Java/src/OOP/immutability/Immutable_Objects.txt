Rules for immutability
 1. Make class final
 2. Male fields private final
 3. Initialize via constructor
 4. No setters
 5. Return objects instead of modifying
 6. Defend against mutable fields (advanced)

    Thread-safe by default
    Safe caching

    Easier debugging

Defend against mutable fields
 🔥 This is the issue:
     - You exposed your internal state by reference
 If your field is mutable, never expose it directly

Defending against mutable fields means preventing external code from modifying an object’s internal state by returning
copies of mutable objects or copying them during construction. This preserves encapsulation and ensures immutability.

| Type                 | Meaning                        |
| -------------------- | ------------------------------ |
| Shallow immutability | container is safe              |
| Deep immutability    | everything inside is also safe |

🧪 Final mental check
Which is better in real systems:
    Make inner objects immutable
    Deep copy everything
👉 And why?

Why making inner objects immutable is better
1. 🚀 Performance (big reason)
      ❌ Deep copying
          👉 Every time:
              You allocate new objects
              You consume more memory
              You add CPU overhead
      ✅ Immutable objects
          👉 No extra object creation
          👉 Safe to share references
          👉 Much faster
2. 🧠 Simplicity
    ❌ Deep copy approach
        More code
        More bugs
        Easy to forget copying somewhere
    ✅ Immutable objects
        Clean
        Predictable
        No hidden side effects

Making inner objects immutable is preferred because it avoids the overhead of deep copying, simplifies design, and
ensures thread safety. It allows safe sharing of objects without risk of unintended mutation, whereas deep copying is
more expensive and error-prone.

🔥 Final mental model

Immutability = design-level safety
Deep copy = defensive workaround

Immutable objects are inherently thread-safe because they can’t be changed after creation
Immutable objects are safe to cache because their state never changes

Immutability improves thread safety by eliminating shared mutable state, making objects inherently safe in concurrent
environments. It enables reliable caching since objects cannot change after creation, and improves performance by
avoiding synchronization overhead and allowing safe reuse of instances.

⚠️ Tradeoff (important)
Immutability can:
 create more objects
BUT:
👉 Modern JVM handles this well (GC optimized)

In Java, objects are allocated in the heap and accessed via references stored in the stack. When objects are shared
across references, mutations affect all references, which can lead to bugs and race conditions. That’s why immutability
and stateless design are important—they prevent shared mutable state and make systems safer and more scalable.

Java does not copy objects by default—it copies references, so multiple variables can point to the same object and share state.
