    🧠 What encapsulation really is
    Most people think:
     "Encapsulation = private fields + getters/setters"
    This surface-level and incomplete X

    Real Definition:
     "Encapsulation is about controlling how data is accessed and modified to protect invariants"
      👉 Key word: invariants (rules that must always hold true)
      👉 You don’t expose the data—you expose safe operations

    Encapsulation is about RESTRICTION, not access
     Bad Mindset:
      "Let me expose everything with getters/setters"

        class User {
            private String email;

            public void setEmail(String email) {
                this.email = email;
            }
        }
        👉 Anyone can set invalid email

     Good Mindset:
      "What should the outside world NOT be allowed to do?"

        class User {
            private String email;

            public void changeEmail(String newEmail) {
                if (!newEmail.contains("@")) {
                    throw new IllegalArgumentException("Invalid email");
                }
                this.email = newEmail;
            }
        }
        👉 Behavior > raw data access

      🧠 3. Encapsulation vs Data Classes
            👉 Object enforces its own rules

      🔥 4. Encapsulation + Immutability (power combo)
             Best designs often:
                 Hide state (private final ..)
                 Make it hard to change (enforce business rules / invariants)
             👉 No setters = safer object

      🧠 6. Encapsulation in real backend systems

          In a Spring Boot system:
              Entities → controlled updates
              Services → enforce rules
              DTOs → transfer data (not behavior)

🔥 7. Golden rules (memorize these)
       1. Don’t expose fields—expose behavior
       2. Don’t add setters unless absolutely necessary
       3. Validate at the boundary (constructor or method)
       4. Make illegal states unrepresentable

“Encapsulation is about protecting the internal state of an object by exposing only controlled operations
that enforce business rules. Instead of exposing setters, we design methods that preserve invariants and prevent invalid states.”

==============================----------------------------------------===========================
1. How do I protect invariants
   - Validate in methods/constructor
   - Don't expose raw fields
   - Use controlled operations

   🔥 Never allow an object to enter an invalid state -- ever
      ~ Enforce at boundaries
      ~ Don't expose dangerous operations
        e.g setBalance()
      ~ Only expose safe transitions
        e.g pay()
      ~ Identity fields
        e.g ID - set once in constructor, No setter
      👉 If someone cannot misuse your class, you’ve protected invariants.

2. When should I NOT expose setters?
   - A setter is bad when it allows someone to break the rules of your object
   - If changing a field can break business logic - DOT NOT use a setter
Don't Use when:
 - The field must follow rules
 - The object has a lifecycle. ( Enforce valid transitions ) e.g CREATED -> PAID -> SHIPPED
 - You want immutability
 - You want thread safety. "Mutable shared state = bugs"

Use when:
 - Field has no strict rules
 - It's simple DTO

🔥 The mental shift (THIS is key)
    Instead of thinking:
     “Should I add a setter?”
    Think:
     “What operations should this object allow?”

“Setters should not be exposed when they can break business rules or invariants. Instead of allowing direct field
modification, we expose methods that enforce valid state transitions and protect the integrity of the object.”
