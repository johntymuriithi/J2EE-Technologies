     Rules in Overriding equals() and hashCode()
    ===============================================

    Rule 1: The Contract (They Must Be Overridden Together)
         Never override one without the other.
         - If obj1.equals(obj2) is true, then obj1.hashCode() must equal obj2.hashCode()
         - If two objects have the same hash code, they do not have to be equal. ( this is hash collision )
         - Consequence: If you break this, your object will disappear or get duplicated inside a HashSet or HashMap.

    Rule 2: Honor the 5 Mathematical Properties of equals()
        Your equals() method must strictly satisfy the contract defined in the java.lang.Object specification:
         - Reflexive: x.equals(x) must always be true;
         - Symmetric: if x.equals(y) is true, then y.equals(x) must be true
         - Transitive: If x.equals(y) and y.equals(z) are both true, then x.equals(z) must be true;
         - Consistent: Multiple invocations must return the same result, provided no field values were 'modified'
         - Null Comparison: x.equals(null) must always return false. (Never let it throw a NullPointerException)

    Rule 3: Use the Exact Same Fields in Both Methods
        Consistency is critical. Any field you use to determine equality in eqauls() must be used to calculate hash in hashCode().
         - Bad: Using id and email in equals(), but only using id in hashCode()
         - Good: if equals() checks id and email, then hashCode() must process both id and email.

    Rule 4: Prefer Immutable Fields (The 'Disappearing Object' Bug)
        If you add an object to a HashSet(), and then modify a field that is used inside hashCode(), the object's hash changes
         - The Bug: The collection will look for the object in a new memory bucket based on the new hash. It will fail to
         find it, making the object permanently "lost" inside the collection
         - Professional Rule: Whenever possible, calculate equality and hash codes using immutable fields (fields marked final)

    Rule 5: Use getClass for Strict Equality, instanceof for Frameworks
        - Use getClass() != obj.getClass() if a subclass should never be considered equal to a parent class. This enforces
        strict type matching
        - Use obj instanceof User if you are using frameworks like Hibernate or JPA. Frameworks often create dynamic proxies
        (subclasses) of your entities at runtime. If you use getClass(), a Hibernate proxy will fail equality checks against
        a normal entity instance

    Rule 6: Never Use Business Fields that Rely on Network or I/O
        Never include fields whose values depend on unreliable external states
        - Example: Including a java.net.URL object in your hash code is an industry anti-pattern. The URL.equals() method
        triggers a blocking network call to resolve the IP address, causing massive performance drops and unpredictable
        bugs if the network goes down.

    When overriding equals, prefer Objects.equals(obj1, obj2) -- is fast, handles nulls safely, no memory leaks
    When overriding hashCode, prefer Object.hash(this.field),It automatically handles null safety and provides a well-distributed hash algorithm

    The Professional Templates (Production-Ready
        Option A: Modern Standard Java (Java 7+)
           @Override
           public boolean equals(Object obj) {
               if (this == obj) return true;
               if (obj == null || getClass() != obj.getClass()) return false;
               User other = (User) obj;
               return Objects.equals(this.name, other.name) &&
                      Objects.equals(this.email, other.email);
           }

           @Override
           public int hashCode() {
               return Objects.hash(name, email); // Automates safe null-handling and distribution
           }

    Option B: Enterprise Java (Records - Java 14+)
        If your object is just a data container, use a Java Record.
        The compiler automatically generates flawless, performant, and immutable implementations of equals() and hashCode() for you under the hood:

        // No manual equals/hashCode code needed! It is fully built-in.
        public record User(String name, String email) {}
