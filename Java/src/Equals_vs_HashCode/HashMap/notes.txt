  In HashMap, keys are OBJECT REFERENCES
    variables of object types hold references,
    not actual objects

  How does a HashMap work internally

    A HashMap depends on BOTH:
        hashCode() → find bucket
        equals() → find exact key inside bucket

      Imagine: map.put("John", 45);
      Later: map.get("John");

      How can Java find "John" FAST without checking every element?
      Without Hashing, Java would scan one by one: Mary? no, Peter? no, John? yes --> O(n) time, slow for millions of records

      Solution: Hashing
      Instead of searching everything:
      Java computes: hashCode("John") say = 99932, the java transforms it into a bucket index, bucket = hash % arraySize
                 99932 % 16 = 7, Store "John" in bucket 7;

  Internal Structure of HashMap
    At core: HashMap = array of buckets
    Think
       0 →
       1 →
       2 →
       3 →
       4 →
       5 →
       6 →
       7 → ("John",95)
       8 →
    HashCode answers "Which bucket should this object go into

  What happens during put()
    map.put(key, value)

     STEP 1 -- Compute hashCode
        hash = key.hashCode()

     STEP 2 -- Find, bucket = hash % capacity

     STEP 3 -- Go to bucket
        bucket 7

     STEP 4 -- Compare keys using equals()
        Because bucket may already contain objects.
        Java checks: existingKey.equals(newKey)

     🔥 VERY IMPORTANT
         hashCode DOES NOT determine equality

         It only determines:
            where to search
         equals determines actual equality
            is this truly the same key?

  Collision
    Two different objects can produce same bucket
      "John" → bucket 7
      "Mike" → bucket 7

  🧠 How Java handles collisions

      Bucket stores multiple entries.
        Internally:
          bucket 7 →
             ("John",95)
             ("Mike",88)
      (Java uses linked list / tree internally depending on size)

  What Happens during get()
    map.get("John")
    1. Compute hasCode, bucket 7
    2. Search only inside bucket 7
    3. Use equals(): existingKey.equals(searchKey)

=========================================================================
    When two values results to the same hashcode:
    Java replaces the existing one by default
        1. If you want to prevent a replacement, Java provides modern Map methods to handle it:
            1. Only insert if missing (putIfAbsent)
            2. Throw an error or handle duplicates manually (merge)
            If you wanted to combine or add the numbers together instead of overwriting, you could do this:
               person.put("John", 23);
               // If "John" exists, add the old value and new value together
               person.merge("John", 29, Integer::sum);

               System.out.println(person); // Prints: {John=52}


    NB
        Summary
         - If the key is the same, Java overwrites the value.
         - If the keys are different but collide in memory, Java chains them in a List / Tree.


======================================================================================================
| Situation              | Bucket Finding   | Equality Check         | Result                      |
| ---------------------- | ---------------- | ---------------------- | --------------------------- |
| Override neither       | Identity-based   | Identity-based         | Works by reference identity |
| Override only equals   | ❌ Wrong bucket   | Never reached          | Retrieval fails             |
| Override only hashCode | ✅ Correct bucket | ❌ Reference comparison | Retrieval fails             |
| Override both          | ✅ Correct bucket | ✅ Logical equality     | Correct behavior            |
