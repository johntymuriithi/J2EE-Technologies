    1. What problem do equals() and hashCode() solve?
       Core Idea ~ They define how java decides

       "Are these two objects logically the same?"
       "Where should I store/find this object in hash-based structures?"

       | Method       | Question it answers                      |
       | ------------ | ---------------------------------------- |
       | `equals()`   | Are these two objects equal in meaning?  |
       | `hashCode()` | Which bucket should this object go into? |

    2. Default behavior
       Every class inherits from Object;

       class Object {
            boolean equals(Object o) { return this == o; }
            int hashCode() { return memory_address; }
       }

       Default behavior = identity comparison

    3. Why override them?
       Because in real systems we care about logical equality, not memory identity

    4. equals() contract
        Rules:
            1. Reflexive
            2. Symmetric
            3. Transitive
            4. Consistent
            5. Null-safe

        If two objects are equal, they must behave consistently everywhere in the system

    5. hashCode()
       Step 1: hashCode decides bucket
               map.put(key, value)
               ~ bucket = hashCode(key)
       Step 2: equals confirms match
               ~ Inside bucket
                 if (equals(existingKey)) --> replace / return

       if equals() is true --> hashCode Must be same

    5. What happens if you Override equals but not hashCode
        ~ HashSet / HashMap breaks consistency

    Why hashCode exists (performance reason)

    hashCode = Where do I look? O(1) average lookup - narrows search "Look in this neighborhood"
    equals = Is it really the same? - confirms identity - Is this the exact house

    “equals() defines logical equality between objects, while hashCode() determines the bucket location in hash-based
    collections. If two objects are equal according to equals(), they must have the same hashCode to maintain consistency
    in data structures like HashMap and HashSet.”

=================================------------------------------------=========================
    1. Default hashCode() ( from Object )
        If you dont override it, then java uses Object.hashCode()

        What does default hashCode represent?, object identity / memory-based identity

        🔥 Important:
            ~ Default hashCode is NOT actual memory address,
              but JVM-derived identity hash.

        2. Overridden hashCode()
           🔥 What does THIS represent?
           Now hashCode represents:
               logical state/data
               NOT object identity.

    | Default hashCode                             | Overridden hashCode             |
    | -------------------------------------------- | ------------------------------- |
    | Based on object identity                     | Based on logical data           |
    | Different objects → usually different hashes | Equal objects → same hash       |
    | From Object class                            | Custom implementation           |
    | Works with `==` semantics                    | Works with `equals()` semantics |

    🧠 Deep insight
    Default behavior
        Java assumes:
            "Objects are only equal if they are literally the same object"
        Overridden behavior

        You are telling Java:
            "These objects should be considered equal based on their contents"
        🔥 This is why equals and hashCode must align

        If you override:
            equals()
            to compare logical data,
            then hashCode must ALSO use same logical data.

    | Concept             | Meaning                           |
    | ------------------- | --------------------------------- |
    | Default hashCode    | “Who am I physically?”            |
    | Overridden hashCode | “What data defines me logically?” |

    🧠 Default equals() vs Overridden equals()
        | Default equals()         | Overridden equals()         |
        | ------------------------ | --------------------------- |
        | Compares object identity | Compares logical state/data |
        | Uses `==` internally     | Uses custom logic           |
        | “Same object?”           | “Same meaning?”             |

        1. Default equals() (from Object)
            Java uses Object.equals() which is basically return this == obj;
            ~ "Are these literally the same object in memory"

            🧠 Important
                Default equals() = identity equality.

        2. Overridden equals()
            "Do these objects contain the same logical data?"

        ⚖️ Big conceptual difference
            Default equals
                Physical identity
            Overridden equals
                Logical equality
            🧠 Real-world analogy
                Default equals()
                    "Are these the exact same passport?"
                Overridden equals()
                    "Do these passports belong to the same person?"
            🔥 Why override equals at all?
                Because business systems usually care about:

                    same email
                    same ID
                    same account number

                    NOT whether objects occupy same memory.

    🧠 Deep insight
        equals() defines the equality rule.
        hashCode() must follow that rule.

    | Method              | Question               |
    | ------------------- | ---------------------- |
    | `==`                | Same object reference? |
    | default equals()    | Same object reference? |
    | overridden equals() | Same logical value?    |

    🔥 One subtle thing many miss
        Even after overriding equals:

            u1 == u2

        still checks reference identity ONLY.
        == is never affected by overriding equals.

    🎯 Final mastery statement

    Hash-based collections depend on stable hashCodes. If fields used in equals() or hashCode() are mutated after
    insertion, the object may become unreachable, duplicated, or inconsistent because it remains stored in the
    old bucket while future lookups use the new hashCode.
